# AI 翻译助手 macOS PRD

版本：0.1 Draft  
日期：2026-06-18  
平台：macOS  
产品形态：菜单栏常驻工具 + 全局快捷键 + 轻量浮窗  
当前范围：MVP 产品定义

## 1. 背景

用户在阅读英文网页、论文、PDF、图片、软件界面、聊天消息或不可复制文本时，经常需要临时翻译。现有翻译产品通常要求用户切换应用、复制文本、打开网页或进入完整翻译工作台，容易打断当前工作心流。

本产品希望把翻译变成一种轻量的系统级能力：用户在任何界面遇到看不懂的内容时，通过快捷键截图或呼出输入窗口，即可快速获得中英文互译结果。

## 2. 产品定位

一句话定位：

> 任何地方，看不懂，按一下，马上懂。

产品关键词：

- 轻量
- 即用即走
- 不打断当前工作流
- 中英文高质量互译
- 用户可配置 LLM 接口
- 隐私边界清晰

第一版不做大而全的翻译工作台，而是优先做好两个高频场景：

1. 截图区域翻译：对屏幕任意区域进行 OCR 识别并翻译。
2. 快捷输入翻译：呼出小窗口，输入或粘贴文本后快速翻译。

## 3. 目标用户

### 3.1 核心用户

- 经常阅读英文资料的中文用户。
- 经常处理中英双语工作的产品、研发、设计、运营和研究人员。
- 需要阅读不可复制文字的用户，例如图片、截图、软件界面、PDF 扫描件。
- 希望自带 LLM API，而不是绑定单一翻译服务的高级用户。

### 3.2 典型场景

- 阅读英文文档、论文或网页时，框选一段内容快速翻译。
- 看英文软件界面、报错信息、图片字幕时，通过截图翻译理解含义。
- 写作、聊天、邮件时，呼出小窗输入中文并翻译成英文。
- 想保留原文和译文对照，用于校对、学习或转述。

## 4. 产品目标

### 4.1 MVP 目标

- 用户可以通过全局快捷键完成截图翻译。
- 用户可以通过全局快捷键呼出即用即走的文本翻译窗口。
- 用户可以配置自己的 OpenAI-compatible LLM 接口。
- 用户可以默认完成中英文自动互译。
- 用户可以查看原文和译文对照，并一键复制译文。

### 4.2 成功指标

- 常规短文本从触发到看到译文的时间尽量控制在 3-5 秒内。
- 核心翻译路径不超过 2 次显式操作。
- 用户完成首次 LLM 配置后，日常使用不需要反复进入设置。
- 截图翻译失败时，用户能明确知道失败原因和下一步操作。

## 5. 非目标

MVP 阶段不优先支持：

- 完整文档翻译工作台。
- 多人协作或团队术语库。
- 原位覆盖翻译的完美视觉还原。
- 浏览器插件生态。
- 复杂翻译记忆库。
- 多模型自动路由。
- 面向专业译员的 CAT 工具能力。

## 6. 产品原则

1. 不打断：默认后台常驻，只在用户主动触发时出现。
2. 快速退出：所有浮窗都应支持 `Esc` 关闭。
3. 键盘优先：高频动作应能通过快捷键完成。
4. 结果可信：优先保留原文对照，避免用户无法校验译文。
5. 隐私透明：默认只发送 OCR 后文本给 LLM，不上传截图图片。
6. 配置清晰：普通用户只看必要配置，高级参数收进高级设置。

## 7. 信息架构

产品由 4 个主要模块组成：

1. 菜单栏入口
2. 截图翻译
3. 快捷文本翻译
4. 设置中心

### 7.1 菜单栏入口

菜单栏图标提供以下入口：

- 快捷文本翻译
- 截图翻译

- 退出应用

### 7.2 截图翻译

用户通过快捷键进入框选模式，对任意屏幕区域进行截图、OCR 和翻译。

### 7.3 快捷文本翻译

用户通过快捷键呼出轻量输入窗口，输入或粘贴文本后直接获得翻译。

### 7.4 设置中心

设置中心包含：

- 默认语言
- 快捷键
- LLM 接口
- 翻译偏好
- 术语表
- 历史记录
- 隐私与安全

## 8. 核心用户流程

### 8.1 首次启动流程

1. 用户打开应用。
2. 应用显示欢迎和必要权限说明。
3. 用户配置 LLM Provider。
4. 用户填写 API Key。
5. 用户选择模型或填写模型名。
6. 用户点击测试连接。
7. 测试成功后，应用提示设置全局快捷键。
8. 用户授予屏幕录制权限和辅助功能权限。
9. 应用进入菜单栏常驻状态。

异常处理：

- API Key 无效：提示认证失败，并展示服务商返回的简短错误。
- 网络失败：提示网络或 Base URL 不可达。
- 权限未授予：标记相关功能不可用，并提供重新打开系统设置的入口。

### 8.2 截图翻译流程

1. 用户按下截图翻译快捷键。
2. 屏幕进入区域框选模式。
3. 用户框选目标区域。
4. 应用截取区域图像。
5. 应用使用本地 OCR 识别文字。
6. 应用根据默认规则识别源语言和目标语言。
7. 应用调用 LLM 翻译 OCR 文本。
8. 应用在截图区域附近展示翻译浮窗。
9. 用户复制译文、切换对照模式、重新翻译或关闭窗口。

默认结果形态：

- 第一版使用对照浮窗。
- 原文和译文按段落对齐展示。
- 用户可以选择只看译文或原文译文对照。

异常处理：

- 未识别到文字：提示“未识别到可翻译文字”，并允许重新框选。
- OCR 结果很短或疑似错误：展示原文，让用户可以手动编辑后重译。
- LLM 请求失败：保留 OCR 原文，并提示失败原因。
- 翻译超时：允许重试或复制 OCR 原文。

### 8.3 快捷文本翻译流程

1. 用户按下文本翻译快捷键。
2. 应用在当前屏幕中央或鼠标附近呼出轻量输入窗。
3. 输入框自动聚焦。
4. 用户输入或粘贴文本。
5. 应用自动判断翻译方向。
6. 用户按 `Enter` 或等待短暂停顿后触发翻译。
7. 应用展示译文。
8. 用户按 `Cmd + Enter` 复制译文并关闭窗口，或按 `Esc` 关闭。

建议快捷键：

- `Enter`：触发翻译或复制当前译文。
- `Cmd + Enter`：复制译文并关闭。
- `Esc`：关闭窗口。
- `Cmd + K`：清空输入。
- `Cmd + Shift + L`：交换源语言和目标语言。

## 10. LLM 配置需求

### 10.1 快速配置

快速配置面向普通用户，只展示必要项：

- 服务商
- API Key
- 模型
- 测试连接

服务商建议：

- OpenAI
- Azure OpenAI
- OpenAI-compatible
- Local compatible endpoint

### 10.2 高级配置

高级配置面向熟悉模型接口的用户：

- Base URL
- Model name
- API Key
- 自定义 Headers
- Timeout
- Temperature
- Max tokens
- System prompt
- Translation prompt template

### 10.3 默认翻译 Prompt 要求

翻译 Prompt 应满足：

- 保留原意。
- 保留段落结构。
- 保留代码、变量名、产品名和专有名词。
- 不额外解释，除非用户选择解释模式。
- 对不确定的翻译保持克制。

示例要求：

```text
你是一个专业翻译助手。请将用户提供的文本翻译为目标语言。
要求：
1. 保留原文段落结构。
2. 保留代码、变量名、链接、产品名和专有名词。
3. 不添加原文不存在的信息。
4. 只输出译文，除非用户要求原文对照或解释。
```

## 11. 语言策略

### 11.1 MVP 默认规则

- 如果输入主要为中文，默认翻译为英文。
- 如果输入主要为英文，默认翻译为中文。
- 如果输入混合中英，默认翻译为用户设置的目标语言。

### 11.2 未来扩展

未来支持：

- 固定目标语言。
- 跟随上次目标语言。
- 每次翻译前手动选择。
- 自动识别更多语言。

## 12. 隐私与安全

### 12.1 默认隐私策略

- 默认不上传截图图片。
- 默认只将 OCR 后文本发送给 LLM。
- API Key 存储在 macOS Keychain。
- 翻译历史默认本地保存，用户可关闭。
- 用户可一键清空历史记录。

### 12.2 需要明确告知用户的内容

- 使用云端 LLM 时，文本会发送到用户配置的服务商。
- 如果用户启用图片理解或原位翻译增强，截图图片可能会被发送到模型服务。
- 本地 OCR 需要屏幕录制权限。
- 全局快捷键可能需要辅助功能权限。

## 13. 权限需求

| 权限 | 用途 | 必要性 |
| --- | --- | --- |
| 屏幕录制 | 截取用户框选区域 | 截图翻译必需 |
| 辅助功能 | 全局快捷键、窗口置顶、读取部分系统状态 | 建议开启 |
| 网络访问 | 调用用户配置的 LLM 服务 | 翻译必需 |
| Keychain | 安全保存 API Key | 必需 |

## 14. 交互与界面要求

### 14.1 视觉原则

- 遵循 macOS 原生质感。
- 窗口轻量，不使用复杂大面板。
- 内容区域优先可读性，不做强装饰。
- 菜单和设置页清晰、直接。
- 错误状态要明确，但不打断用户。

### 14.2 截图翻译浮窗

浮窗包含：

- 源语言和目标语言。
- 原文区域。
- 译文区域。
- 复制译文按钮。
- 复制原文按钮。
- 重新翻译按钮。
- 关闭按钮。
- 对照模式切换。

浮窗行为：

- 默认展示在截图区域右侧或下方。
- 如果空间不足，自动调整到可见区域。
- 失焦后不立即关闭，避免用户还没复制结果。
- 支持 `Esc` 关闭。

### 14.3 快捷文本翻译窗

窗口包含：

- 输入区。
- 目标语言选择。
- 翻译结果区。
- 复制按钮。
- 清空按钮。
- 设置入口。

窗口行为：

- 呼出后自动聚焦输入框。
- 粘贴文本后可自动翻译，也可由用户按键触发。
- 复制并关闭是最高频操作，应被优化。
- 窗口关闭后不保留前台打扰。

### 14.4 设置页

设置页分组：

- General：默认语言、启动项、菜单栏行为。
- Shortcuts：截图翻译、文本翻译。
- Model：Provider、API Key、模型、测试连接。
- Translation：风格、Prompt、术语表。
- Privacy：历史记录、上传策略、清空数据。
- About：版本、反馈入口。

## 15. 技术建议

### 15.1 推荐技术路线

第一版推荐使用原生 macOS：

- SwiftUI：设置页、基础窗口。
- AppKit：菜单栏、全局快捷键、浮窗、截图框选。
- Vision Framework：本地 OCR。
- Keychain：API Key 存储。
- URLSession：调用 LLM API。

理由：

- macOS 系统能力集成更稳定。
- 权限体验更自然。
- 截图、菜单栏、悬浮窗和快捷键更容易打磨。
- 第一版只做 macOS，不需要为跨平台付出复杂度。

### 15.2 可选技术路线

如果后续明确需要跨平台，可以考虑 Tauri：

- 前端：React 或 Svelte。
- 桌面壳：Tauri。
- macOS 原生能力通过插件或 Swift sidecar 实现。

但 MVP 不建议优先跨平台，因为截图框选、OCR、快捷键、权限和浮窗行为在不同系统差异明显。

## 18. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| OCR 识别不稳定 | 翻译结果错误 | 展示 OCR 原文，允许编辑后重译 |
| LLM 延迟过高 | 即用即走体验变差 | 增加超时、加载状态和快速重试 |
| 权限授予复杂 | 首次启动流失 | 提供清晰权限解释和系统设置入口 |
| API 配置门槛高 | 普通用户难以上手 | 提供快速配置和测试连接 |
| 原位覆盖复杂 | 实现成本高 | MVP 使用对照浮窗，后续再做 |
| 历史记录隐私担忧 | 用户不信任 | 默认本地保存，可关闭和清空 |

## 20. 待确认问题

1. 第一版是否只面向个人用户，还是需要考虑团队分发？
2. 是否需要内置默认模型服务，还是完全由用户自带 API Key？
3. 是否允许保存翻译历史，默认开启还是默认关闭？
4. 目标用户更偏研发/产品，还是更泛办公用户？
5. 是否需要 App Store 分发，还是先用独立安装包？
6. 是否需要支持离线 OCR 和离线翻译两个层级？

## 21. MVP 结论

第一版应聚焦在最短路径：

1. 用户按快捷键。
2. 用户截图或输入文本。
3. 应用显示原文和译文。
4. 用户复制结果并回到原来的工作。

这个产品的关键不是提供最多翻译功能，而是把“临时理解一段陌生语言”这件事做到足够快、足够轻、足够不打扰。
